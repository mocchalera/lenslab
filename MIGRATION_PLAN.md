Status: PHASE 1 planning only. No implementation changes in this phase.

Last verified: 2026-04-22 JST

Official sources checked:

- OpenAI Images guide: https://developers.openai.com/api/docs/guides/image-generation

## 1. Current State

LensLab is a Vite + React 19 + TypeScript app. The current generation path is direct client-side Gemini access:

- `App.tsx`
  - imports `generateSimulation` from `./services/geminiService`
  - calls `await generateSimulation(file, params)` inside `handleGenerate`
  - expects a display-ready data URL string and stores it as `generatedImage` / `HistoryItem.imageUrl`
- `services/geminiService.ts`
  - reads `process.env.API_KEY`
  - Vite maps `process.env.API_KEY` and `process.env.GEMINI_API_KEY` to `env.GEMINI_API_KEY`
  - instantiates `new GoogleGenAI({ apiKey })`
  - uses model `gemini-2.5-flash-image`
- `README.md`
  - documents only `GEMINI_API_KEY`

Current Gemini input/output signature:

```ts
export const generateSimulation = async (
  imageFile: File,
  params: SimulationParams
): Promise<string>
```

Input:

- `imageFile: File`
  - converted with `FileReader.readAsDataURL`
  - stripped to raw base64
  - passed to Gemini as:

```ts
{
  inlineData: {
    data: string;
    mimeType: string;
  }
}
```

- `params: SimulationParams`
  - camera, lens, aperture, exposure, lighting, scene, clothing, pose, output profile, fidelity
  - expanded into one large prompt in `geminiService.ts`

Gemini request shape:

```ts
await ai.models.generateContent({
  model: "gemini-2.5-flash-image",
  contents: {
    parts: [imagePart, { text: prompt }],
  },
});
```

Output:

- iterates `response.candidates[0].content.parts`
- returns the first `part.inlineData` as:

```ts
`data:${part.inlineData.mimeType};base64,${part.inlineData.data}`
```

Failure behavior:

- missing key throws `Error("API Key is missing.")`
- no inline image throws `Error("No image generated by the simulation engine.")`
- API errors are logged as `"Simulation failed:"` and rethrown
- `App.tsx` displays `err.message || "Simulation failed. Please try again."`

## 2. OpenAI Images API Facts

The user requested "OpenAI Images 2.0 (gpt-image-1 series)", but official docs checked on 2026-04-22 identify the latest GPT Image API model for ChatGPT Images 2.0-era image generation as `gpt-image-2`. The docs also list prior/current GPT Image family models including `gpt-image-1.5`, `gpt-image-1`, and `gpt-image-1-mini`.

Recommendation for PHASE 2:

- default OpenAI model: `gpt-image-2`
- keep the adapter type open to `gpt-image-1.5`, `gpt-image-1`, and `gpt-image-1-mini` if the user explicitly wants the older family or needs lower cost
- expose the model choice internally first; UI should switch provider only unless model switching is approved

Relevant official endpoints:

- Generate from text:
  - `POST https://api.openai.com/v1/images/generations`
  - SDK equivalent: `client.images.generate(...)`
- Edit / transform using input image references:
  - `POST https://api.openai.com/v1/images/edits`
  - SDK equivalent: `client.images.edit(...)`
- Variations:
  - API reference still has `POST /v1/images/variations`, but the guide says variations are for models that support them, such as DALL-E 2. Do not rely on variations for GPT Image models.

LensLab's current workflow is image-to-image transformation, so OpenAI provider should use `POST /v1/images/edits`, not `/v1/images/generations`.

Minimum raw HTTP request plan:

```ts
const form = new FormData();
form.append("model", "gpt-image-2");
form.append("image[]", imageFile, imageFile.name || "source.png");
form.append("prompt", prompt);
form.append("quality", "medium");
form.append("size", "auto");
form.append("output_format", "png");

const response = await fetch("https://api.openai.com/v1/images/edits", {
  method: "POST",
  headers: {
    Authorization: `Bearer ${openAIKey}`,
  },
  body: form,
});
```

Expected response shape for GPT Image models:

```ts
interface OpenAIImagesResponse {
  created: number;
  data?: Array<{
    b64_json?: string;
    revised_prompt?: string;
    url?: string;
  }>;
  output_format?: "png" | "webp" | "jpeg";
  quality?: "low" | "medium" | "high" | "auto";
  size?: string;
  usage?: {
    input_tokens: number;
    input_tokens_details?: {
      image_tokens?: number;
      text_tokens?: number;
    };
    output_tokens: number;
    total_tokens: number;
  };
}
```

Output normalization:

```ts
const mimeType = `image/${response.output_format ?? "png"}`;
const dataUrl = `data:${mimeType};base64,${response.data?.[0]?.b64_json}`;
```

Key official parameters to support in the adapter:

- `model`: `gpt-image-2` recommended; also allow `gpt-image-1.5`, `gpt-image-1`, `gpt-image-1-mini` by type
- `image[]`: one or more input image files for edit/reference workflows
- `prompt`: generated LensLab optical prompt
- `quality`: `low`, `medium`, `high`, `auto`
- `size`:
  - `gpt-image-2`: flexible sizes, with common values including `1024x1024`, `1536x1024`, `1024x1536`, larger 2K/4K sizes, or `auto`
  - older GPT Image models: plan for common fixed values only unless docs are rechecked at implementation time
- `output_format`: `png`, `jpeg`, `webp`
- `output_compression`: `0` to `100`, only for JPEG/WebP
- `background`: `opaque` or `auto` for `gpt-image-2`; do not use `transparent` with `gpt-image-2`
- `moderation`: `auto` or `low`
- `input_fidelity`:
  - omit for `gpt-image-2`; docs state it is always high fidelity for image inputs
  - if using `gpt-image-1`, consider `input_fidelity: "high"` only after confirming current API compatibility

## 3. Common Provider Interface

Create `services/imageProvider.ts` in PHASE 2. It should hide provider-specific file conversion, response parsing, latency measurement, cost metadata, and user-facing error text.

Proposed TypeScript types:

```ts
import { SimulationParams } from "../types";

export type ImageProviderId = "gemini" | "openai";

export type OpenAIImageModel =
  | "gpt-image-2"
  | "gpt-image-1.5"
  | "gpt-image-1"
  | "gpt-image-1-mini";

export type ImageOutputFormat = "png" | "jpeg" | "webp";
export type ImageQuality = "low" | "medium" | "high" | "auto";

export interface ImageGenerationRequest {
  imageFile: File;
  params: SimulationParams;
  providerOptions?: {
    openaiModel?: OpenAIImageModel;
    quality?: ImageQuality;
    size?: string;
    outputFormat?: ImageOutputFormat;
    outputCompression?: number;
    moderation?: "auto" | "low";
  };
}

export interface ImageUsage {
  inputTokens?: number;
  imageInputTokens?: number;
  textInputTokens?: number;
  outputTokens?: number;
  totalTokens?: number;
}

export interface ImageGenerationResult {
  dataUrl: string;
  provider: ImageProviderId;
  model: string;
  mimeType: string;
  latencyMs: number;
  usage?: ImageUsage;
  costHint?: string;
}

export type ImageProviderErrorCode =
  | "missing_api_key"
  | "bad_request"
  | "auth"
  | "rate_limited"
  | "moderation"
  | "no_image"
  | "network"
  | "unknown";

export class ImageProviderError extends Error {
  constructor(
    public readonly provider: ImageProviderId,
    public readonly code: ImageProviderErrorCode,
    message: string,
    public readonly cause?: unknown
  ) {
    super(message);
    this.name = "ImageProviderError";
  }
}

export interface ImageProvider {
  id: ImageProviderId;
  label: string;
  generateSimulation(request: ImageGenerationRequest): Promise<ImageGenerationResult>;
}
```

Compatibility shim:

```ts
export async function generateSimulationWithProvider(
  providerId: ImageProviderId,
  imageFile: File,
  params: SimulationParams
): Promise<ImageGenerationResult>;
```

`App.tsx` can initially use `.dataUrl` and keep existing history behavior. A later phase can display `latencyMs`, `model`, and `usage`.

Provider files:

- `services/geminiService.ts`
  - keep existing `generateSimulation(imageFile, params): Promise<string>` to avoid breaking imports
  - add `geminiProvider: ImageProvider` wrapper around the existing function
  - optionally move shared prompt builder later, but avoid refactor in the first implementation cut
- `services/openaiService.ts`
  - implement `openaiProvider: ImageProvider`
  - build the same LensLab prompt, ideally by reusing an exported prompt builder if we split it from Gemini safely
  - use `fetch` + `FormData` unless SDK is approved
- `services/imageProvider.ts`
  - types, provider registry, error normalization helpers

## 4. SDK vs Fetch

Option A: raw `fetch` with `FormData` (recommended for first implementation)

- no new runtime dependency
- works in Vite/browser with the same direct-client style as current Gemini path
- easy to keep request shape explicit
- must manually type response/error JSON
- still exposes `OPENAI_API_KEY` in the browser bundle if mapped through Vite

Option B: `openai` SDK

- typed client and helper utilities like `toFile`
- adds dependency and bundle weight
- browser usage still exposes the API key and typically requires explicit browser allowance
- Node-oriented examples do not map cleanly to this app's current pure-client architecture

Option C: backend/proxy

- best production security posture
- keeps API keys off the client
- larger architecture change, not requested for PHASE 2
- would alter deployment assumptions beyond the current Vite-only app

Recommendation: use raw `fetch` in PHASE 2, document that direct browser keys are suitable only for local/private use, and leave a proxy as a later hardening task.

## 6. Breaking Differences and Absorption Plan

Image input format:

- Gemini: inline base64 part `{ inlineData: { data, mimeType } }`
- OpenAI: multipart `FormData` with `image[]` file parts for `/v1/images/edits`
- Absorption: providers own file serialization; `App.tsx` passes the original `File` only

Prompt reuse:

- Gemini prompt builder is embedded in `geminiService.ts`
- OpenAI needs the same optical prompt
- Absorption: PHASE 2 should first extract prompt construction into `services/simulationPrompt.ts`, then have both providers consume it. This is the only shared-code refactor needed.

Response format:

- Gemini: generated image is `part.inlineData.data` plus `part.inlineData.mimeType`
- OpenAI: GPT Image models return `data[0].b64_json`; URL output is unsupported for GPT Image models
- Absorption: normalize to `ImageGenerationResult.dataUrl`

Editing vs generation:

- Current Gemini call sends image + prompt in a single content generation call
- OpenAI should use `/v1/images/edits` because LensLab transforms an uploaded source image
- Absorption: OpenAI provider selects edits endpoint internally; no UI distinction

Variations:

- OpenAI variations are not a GPT Image feature to rely on
- Current LensLab does not use variations
- Absorption: omit variations from common interface until there is a separate product requirement

Masks:

- OpenAI edits can accept masks, but LensLab currently has no mask UI
- Absorption: keep `maskFile?: File` out of the initial common interface unless needed later

Output MIME and filenames:

- Current history/download assumes PNG filenames
- OpenAI can return PNG/JPEG/WebP
- Absorption: default OpenAI `output_format` to `png` for compatibility. If JPEG/WebP is later enabled, update download extension logic with a `mimeType -> extension` helper.

Cost and latency:

- Gemini path currently has no usage metadata
- OpenAI GPT Image responses can include token usage; cost and latency vary with size, quality, and image input tokens
- Official docs note complex prompts can take up to about 2 minutes
- Absorption: provider layer records `latencyMs`; OpenAI maps `usage` when present; UI can remain unchanged initially

Moderation / verification:

- OpenAI GPT Image models may require organization verification
- prompts and generated images are moderated
- Absorption: normalize 401/403/429/400 responses into `ImageProviderError` codes and show concise user-facing messages

Environment variables:

- Gemini currently uses `GEMINI_API_KEY` mapped to `process.env.API_KEY`
- OpenAI should use `OPENAI_API_KEY`
- Absorption: add only `process.env.OPENAI_API_KEY` to `vite.config.ts` in PHASE 2. Do not remove or rename Gemini mappings.

## 7. PHASE 2 Implementation Order

1. Extract prompt construction from `services/geminiService.ts` to `services/simulationPrompt.ts`.
2. Add `services/imageProvider.ts` with shared types, registry, and normalized errors.
3. Wrap the existing Gemini function as a provider without changing its public `generateSimulation` return behavior.
4. Add `services/openaiService.ts` using `fetch` + `FormData` against `/v1/images/edits`.
5. Add `OPENAI_API_KEY` mapping in `vite.config.ts` while preserving existing Gemini mappings.
6. Add provider selector state/UI in `App.tsx`.
7. Update README with env vars, switching, and direct-client security limitations.
8. Run `npm run build`.

## 8. Risks / Items Requiring Approval

- Model naming mismatch: the request says "gpt-image-1 series", but official docs now point to `gpt-image-2` for the latest Images API. Approval needed: use `gpt-image-2` as OpenAI default?
- Client-side API key exposure: existing Gemini path already injects a key into the browser bundle. Adding OpenAI the same way is minimal but not production-secure.
- Organization verification: OpenAI GPT Image models may fail with a 403-style error until the account/org is verified.
- Latency: OpenAI high-quality image edits can be slow. Default should likely be `quality: "medium"` and `size: "auto"` or `1024x1536`/`1536x1024` after visual testing.
- Source-image preservation: `gpt-image-2` always uses high-fidelity image input; older `gpt-image-1` may need `input_fidelity: "high"`, but this should be rechecked if the older model is selected.
- Prompt behavior: Gemini and GPT Image will interpret the same optical prompt differently. Visual QA should compare identity preservation, pose/clothing changes, and lens/rendering fidelity.
- MIME handling: defaulting to PNG avoids immediate history/download changes, but JPEG/WebP may be better for latency and cost later.
- Browser CORS/API behavior: official examples use both SDK and curl. Raw browser fetch should be smoke-tested against the actual OpenAI endpoint during PHASE 2 before assuming deploy readiness.
